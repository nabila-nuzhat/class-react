- Rendering vs Loading in SPAs
    - two distinct phases in a Single Page Application's lifecycle:
    - react re-renders 2 reason:
        1. state change
        2. props change
    - 1st renders then side effect happen

    Loading:
        Loading refers to fetching the resources and data needed to display something.

        Downloading JS/CSS/HTML bundles from the server
        Making API calls to fetch data
        Resolving async dependencies (fonts, images, etc.)

        Loading is an I/O-bound operation — the app is waiting on the network or disk.

    Rendering
        Rendering refers to converting data/state into visible UI.

        React/Vue/Angular diffing the virtual DOM and updating the real DOM
        The browser painting pixels on screen (layout → paint → composite)
        Re-rendering components when state changes

        Rendering is a CPU-bound operation — the app is doing computation.

        The Core Distinction

            Loading = waiting for data/resources to arrive.
            Rendering = turning what you have into pixels.

            You can have fast loading but slow rendering (heavy DOM, expensive components),
            or fast rendering but slow loading (large bundles, slow APIs). 
            Optimizing an SPA usually means profiling both separately.

        Diffing (short for "differencing") is the process of comparing two versions of something to find what changed.

- Vite
    Vite is a modern frontend build tool created by Evan You (the creator of Vue.js). The name means "fast" in French — and that's exactly what it's built for.

    Vite is a development environment provider — it gives you a fast, smart workspace to write and test your code, then packages it for the real world when you're ready.

    Vite (French word for "quick", pronounced /viːt/, like "veet") is a build tool that aims to provide a faster and leaner development experience for modern web projects. It consists of two major parts:

        A dev server that provides rich feature enhancements over native ES modules, for example extremely fast Hot Module Replacement (HMR).

        A build command that bundles your code with Rolldown, pre-configured to output highly optimized static assets for production.

            The Two Roles of Vite
                1. During Development → Acts as a Dev Server

                It spins up a local server (usually http://localhost:5173)
                When your browser requests a file, Vite transforms and serves it on the fly
                It watches your files for changes and instantly reflects updates in the browser (HMR — Hot Module Replacement)
                You still write all the code yourself

                2. During Production → Acts as a Bundler

                When you run npm run build, Vite packages all your code into optimized, browser-ready files
                Minifies JS/CSS, splits code, removes unused code (tree-shaking)
                Outputs a dist/ folder you deploy to a server


                
            What it does
            Vite serves two main purposes:

            Development server — serves your source files instantly using native ES modules in the browser, so there's no bundling step during development. This makes startup nearly instantaneous, regardless of project size.
            Production bundler — uses Rollup under the hood to bundle your code for production, with optimizations like tree-shaking, code splitting, and minification.

            Why it's popular
            Traditional bundlers like Webpack bundle everything before the dev server starts. As projects grow, this gets slow. Vite sidesteps this by:

            No bundling on dev startup — it only transforms files on demand, when the browser requests them.
            Hot Module Replacement (HMR) that stays fast even in large apps, since updates are scoped to the changed module only.

            Key features

            Works with React, Vue, Svelte, Vanilla JS, TypeScript, and more out of the box
            Plugin system compatible with Rollup plugins
            Built-in support for CSS, JSON, static assets, and CSS preprocessors
            First-class TypeScript support
            Easy environment variable handling (.env files)

            Basic usage
            bashnpm create vite@latest my-app -- --template react
            cd my-app
            npm install
            npm run dev
            In short, Vite has become the go-to dev tool for modern frontend projects because it's dramatically faster than older bundlers, especially during development.

- States
    - useState — adds local state to a component
    - Local state is data that belongs to a single component and is managed entirely within it — other components 
    can't directly access or modify it.

    - Local state vs. other types of state:

        Local state (useState) — one component only
        Lifted state — moved to a parent and passed down as props
        Global state (Context, Redux, Zustand) — shared across many components

        As a rule of thumb: keep state as local as possible, and only lift or globalize it when multiple components genuinely need it.

- useEffect()
- useEffect dependency array: 
    - syntax: useEffect(() => {
                // Effect code here (runs after render)
            }, [dependencies]);


            useEffect(() => {
            // effect
            }, [dependency1, dependency2]);


    - // 1. No array → runs after EVERY render
            useEffect(() => {
            console.log("runs every time");
            });

        // 2. Empty array [] → runs ONCE (on mount only)
            useEffect(() => {
            console.log("runs once");
            }, []);

        // 3. With values → runs when those values change
            useEffect(() => {
            console.log("runs when count changes");
            }, [count]);

        - Why an Array Specifically?
            Arrays are ordered and iterable — React needs to loop through each dependency and compare it with the previous render's values using Object.is() (strict equality). An array makes this straightforward
