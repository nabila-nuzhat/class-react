    - React itself is not a SPA (Single Page Application) — it's a JavaScript library for building user interfaces. However, the two are closely associated because React is the most popular tool used to build SPAs. Here's the distinction:

    - React vs SPA
        React is a library, not an architecture. It handles the "view" layer — rendering components efficiently using a virtual DOM. It doesn't dictate routing, data fetching, or app structure.

    - SPA is an architectural pattern where the browser loads a single HTML file and JavaScript dynamically updates the content without full page reloads.

    - Why React is associated with SPAs
        - Virtual DOM — React efficiently updates only the parts of the page that change, which is the core mechanic SPAs rely on.

        - Client-side rendering (CSR) — By default, React renders content in the browser using JavaScript, meaning the server sends one HTML shell and React takes over — exactly how SPAs work.
        - Paired with client-side routers — Libraries like React Router simulate page navigation without reloading, completing the SPA experience.

        - Create React App / Vite defaults — The most common React boilerplates generate a single index.html with a root <div>, which is the classic SPA setup.

        - React is NOT always a SPA
          React can be used in many non-SPA ways

- Routing (claude): 
    - Routing is the mechanism that determines how an application responds to a specific URL or path. It maps URLs to specific content, pages, or functions.
    There are two types:
    1. Server-side routing — the browser requests a new page from the server on every URL change. The server processes the URL and returns a full HTML page. Traditional websites work this way.
    2. Client-side routing — the browser handles URL changes without requesting a new page from the server. JavaScript intercepts navigation and swaps content dynamically. This is how modern Single Page Applications (SPAs) work.

- Rendering vs Loading in SPAs
    - two distinct phases in a Single Page Application's lifecycle:
    - react re-renders 2 reason:
        1. state change
        2. props change
    - 1st renders then side effect happen

    Loading:
        Loading refers to fetching the resources and data needed to display something.

        Downloading JS/CSS/HTML bundles from the server
        Making API calls to fetch data
        Resolving async dependencies (fonts, images, etc.)

        Loading is an I/O-bound operation — the app is waiting on the network or disk.

    Rendering
        Rendering refers to converting data/state into visible UI.

        React/Vue/Angular diffing the virtual DOM and updating the real DOM
        The browser painting pixels on screen (layout → paint → composite)
        Re-rendering components when state changes

        Rendering is a CPU-bound operation — the app is doing computation.

        The Core Distinction

            Loading = waiting for data/resources to arrive.
            Rendering = turning what you have into pixels.

            You can have fast loading but slow rendering (heavy DOM, expensive components),
            or fast rendering but slow loading (large bundles, slow APIs). 
            Optimizing an SPA usually means profiling both separately.

        Diffing (short for "differencing") is the process of comparing two versions of something to find what changed.

- Vite
    Vite is a modern frontend build tool created by Evan You (the creator of Vue.js). The name means "fast" in French — and that's exactly what it's built for.

    Vite is a development environment provider — it gives you a fast, smart workspace to write and test your code, then packages it for the real world when you're ready.

    Vite (French word for "quick", pronounced /viːt/, like "veet") is a build tool that aims to provide a faster and leaner development experience for modern web projects. It consists of two major parts:

        A dev server that provides rich feature enhancements over native ES modules, for example extremely fast Hot Module Replacement (HMR).

        A build command that bundles your code with Rolldown, pre-configured to output highly optimized static assets for production.

            The Two Roles of Vite
                1. During Development → Acts as a Dev Server

                It spins up a local server (usually http://localhost:5173)
                When your browser requests a file, Vite transforms and serves it on the fly
                It watches your files for changes and instantly reflects updates in the browser (HMR — Hot Module Replacement)
                You still write all the code yourself

                2. During Production → Acts as a Bundler

                When you run npm run build, Vite packages all your code into optimized, browser-ready files
                Minifies JS/CSS, splits code, removes unused code (tree-shaking)
                Outputs a dist/ folder you deploy to a server


                
            What it does
            Vite serves two main purposes:

            Development server — serves your source files instantly using native ES modules in the browser, so there's no bundling step during development. This makes startup nearly instantaneous, regardless of project size.
            Production bundler — uses Rollup under the hood to bundle your code for production, with optimizations like tree-shaking, code splitting, and minification.

            Why it's popular
            Traditional bundlers like Webpack bundle everything before the dev server starts. As projects grow, this gets slow. Vite sidesteps this by:

            No bundling on dev startup — it only transforms files on demand, when the browser requests them.
            Hot Module Replacement (HMR) that stays fast even in large apps, since updates are scoped to the changed module only.

            Key features

            Works with React, Vue, Svelte, Vanilla JS, TypeScript, and more out of the box
            Plugin system compatible with Rollup plugins
            Built-in support for CSS, JSON, static assets, and CSS preprocessors
            First-class TypeScript support
            Easy environment variable handling (.env files)

            Basic usage
            bashnpm create vite@latest my-app -- --template react
            cd my-app
            npm install
            npm run dev
            In short, Vite has become the go-to dev tool for modern frontend projects because it's dramatically faster than older bundlers, especially during development.

            - Scaffolding Your First Vite Project
            Vite is a modern frontend build tool that provides an extremely fast development experience
            "Scaffolding" means generating a starter project structure automatically.

    - developer friendly

    - CSS module override behavior:
        - CSS Modules vs Global CSS:
            - App.css is a global stylesheet — its styles are injected into the page and apply to any matching element anywhere in the DOM.
            
            - App.module.css is a CSS Module — at build time, class names get transformed into unique hashed identifiers (e.g., .title → .App_title__3fK2m), scoped to the component.

            - Webpack/bundlers typically inject CSS Modules after global CSS, so module styles appear later in the <style> tags → they win.
        
            - CSS Modules don't have any inherent priority — Module CSS win because of bundle injection order, not because of any special browser rule. The behavior can change if your bundler config changes.

        - "CSS specificity" is the algorithm browsers use to decide 
            which CSS rule wins when multiple rules target the same element.
    - Definition:
        - CSS Modules is a CSS file in which all class names are locally scoped by default — meaning styles only apply to the component that imports them, not globally.

        - The "build tool" transforms class names into unique hashes at compile time: Build tool transforms .module.css class names into unique hashes. the hashing happens at build time, not in the browser.

- Build tool VS Dev tool:
    - "build tool", not browser dev tool.
    - These are two different things:

    - Build tool (e.g. Webpack, Vite) — runs during compilation, before the code reaches the browser. It's what transforms .module.css class names into unique hashes.
    - Browser DevTools — runs in the browser after the page loads, used for inspecting and debugging.
    So the hashing happens at build time, not in the browser.

- useEffect()
- useEffect dependency array: 
    - syntax: useEffect(() => {
                // Effect code here (runs after render)
            }, [dependencies]);


            useEffect(() => {
            // effect
            }, [dependency1, dependency2]);


    - // 1. No array → runs after EVERY render
            useEffect(() => {
            console.log("runs every time");
            });

        // 2. Empty array [] → runs ONCE (on mount only)
            useEffect(() => {
            console.log("runs once");
            }, []);

        // 3. With values → runs when those values change
            useEffect(() => {
            console.log("runs when count changes");
            }, [count]);

        - Why an Array Specifically?
            Arrays are ordered and iterable — React needs to loop through each dependency and compare it with the previous render's values using Object.is() (strict equality). An array makes this straightforward

    - A dependency is an external code bundle that your software project needs to function correctly.Instead of writing every single feature from scratch, you reuse code written by other developers.
    ⚙️ How It WorksThe Concept: Your project "depends" on this code to run or build.The Source: You download these dependencies from registries like npm.The Manifest: They are listed inside your project's package.json file.
    👥 The Two Main Types of Dependencies:
        Production Dependencies (dependencies): Code required to run the app in the real world (e.g., Redux, React, Lodash).

        Development Dependencies (devDependencies): Code only needed while building or testing the app (e.g., compilers, linters, test runners).
